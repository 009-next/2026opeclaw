# Development Rules & Vibe Coding Guidelines

## 1. Vibe Coding Principles

- ユーザーの指示に対して、許可を求めるのではなく、自律的に最適なコードを書いて提案・実行してください。
- エラーが発生した場合は、ログを分析し、ユーザーに解決策を尋ねる前に自己修復を試みてください。

## 2. Code Quality

- 全てのコードはTypeScriptで記述し、厳格な型定義（`strict: true`）を行ってください。
- コンポーネントは小さく分割し、再利用性を高めてください。
- コメントはデフォルトで書かない。WHYが自明でない場合（隠れた制約、微妙な不変条件、特定バグへの回避策）のみ1行で記述。

## 3. UI/UX Standard

- 建設現場の担当者がタブレットやPCで操作することを想定し、ボタンや入力フォームは大きく、視認性の高いデザインにしてください。
- Excel/PDFのアップロード時には、ドラッグ＆ドロップ領域と進捗バー（ローディングアニメーション）を実装してください。

## 4. Error Handling

- MCP連携やLLM API呼び出しの失敗時、パース不可能なPDFがアップロードされた時には、技術的なエラーではなく「ユーザーがどうアクションすべきか」を日本語で画面に表示してください。

# 開発ルール（厳守）

1. 1回の応答で1機能に集中（Atomic commits的な考え方）
2. 最初にファイル構造を提案し、承認を得てから実装開始
3. 3Dモデルは最初はシンプルなBox/Cylinderで表現 → 後で高品質モデルに置き換え可能にする
4. 状態は可能な限りZustandで一元管理
5. エラーハンドリングとローディング状態は必ず実装
6. ユーザーがアップロードした工程表のカラム名は柔軟にマッピング可能にする
7. 日本語UIを標準とする（i18n準備も）
8. 機密情報（工事データ）はlocalStorage or IndexedDBにのみ保存（サーバー不要）
   - プロジェクトデータ → IndexedDB（`idb`）
   - APIキー・モデル設定 → localStorage（`settingsStorage`）
9. LLM APIキーはブラウザUIから設定可能（`SettingsPanel` + `useSettings`）。環境変数（`.env.local`）はフォールバックとして機能する
10. 全LLM呼び出しは Next.js API Route 経由でサーバーサイドで実行する。ブラウザから直接LLM APIを呼ばない
